# 注释检查规则

## 检查三项核心指标

### ① 注释密度检查 — 10 行代码至少 3 行注释

**要求：** 每 10 行代码正文（不含空行、括号行、import/require 行）中，至少应有 3 行是注释。

**不达标的表现：**
```ts
// ❌ 反例 — 20 行代码只有 1 行注释
function calcTotal(items) {
  let total = 0
  for (const item of items) {
    total += item.price * item.qty
  }
  if (total > 100) {
    total *= 0.9          // 满 100 打 9 折
  }
  return total
}
```

**达标的表现：**
```ts
// ✅ 正例 — 每段逻辑都有注释说明
/**
 * 计算购物车总金额
 * @param items - 商品列表，每项含 price(单价) 和 qty(数量)
 * @returns 折后总金额
 */
function calcTotal(items) {
  let total = 0
  // 遍历商品，累加金额
  for (const item of items) {
    total += item.price * item.qty
  }
  // 满 100 打 9 折
  if (total > 100) {
    total *= 0.9
  }
  return total
}
```

### ② 注释准确性检查 — 注释必须与代码匹配

**常见问题：**
- **注释过时** — 代码改了但注释没改，注释描述的功能与实际不符
- **注释错误** — 注释说"A"，代码做的是"B"
- **注释无意义** — 注释只是重复代码本身，没有提供额外信息

```ts
// ❌ 反例 — 注释与代码不匹配 / 无意义
let a = 10      // 定义变量 a（废话，没提供额外信息）
a = a + 5       // 增加用户积分（实际上是单纯数字运算，跟积分无关）

// ✅ 正例 — 注释说明意图，不重复代码
let userScore = 10            // 当前用户积分
userScore += 5                // 每日签到奖励 +5 分
```

### ③ 小白视角检查 — 注释是否容易理解

**检查维度：**

| 维度 | 合格 ✅ | 不合格 ❌ |
|------|---------|----------|
| **术语解释** | 专业术语有解释或上下文可推断 | 直接抛术语不解释 |
| **业务目的** | 说清"为什么这样做" | 只说了"做了什么" |
| **变量意图** | 变量名 + 注释让用途一目了然 | 只有缩写或无注释 |
| **复杂逻辑** | 分段说明每步的意图 | 一堆代码没有注释 |
| **边界条件** | 特殊值、异常情况有说明 | 边界情况没有注释 |

```ts
// ❌ 反例 — 小白看不懂
// BST 中序后继
function f(n) {
  if (n.r) return m(n.r)
  let p = n.p
  while (p && n === p.r) { n = p; p = p.p }
  return p
}

// ✅ 正例 — 小白也能懂
/**
 * 在二叉搜索树中查找指定节点的"下一个"节点（中序遍历顺序）
 * 使用场景：从小到大遍历树时，想知道当前节点后面的节点是谁
 * @param node - 当前节点
 * @returns 下一个节点，如果没有则返回 null
 */
function findInorderSuccessor(node) {
  // 情况一：如果有右子树，取右子树中最左边的节点
  if (node.right) return findLeftmost(node.right)
  // 情况二：没有右子树，往上找第一个可以作为"左孩子"的祖先
  let parent = node.parent
  while (parent && node === parent.right) {
    node = parent
    parent = parent.parent
  }
  return parent
}
```

## 检查流程

### 步骤 1：确定检查范围

用户指定要检查的文件或目录，例如：
- `src/stores/` — 检查整个目录
- `src/main/index.ts` — 检查单个文件
- `src/renderer/src/views/` — 检查视图目录

### 步骤 2：逐文件分析

对每个文件执行以下三遍检查：

**第一遍 — 统计注释密度：**
1. 统计文件总行数
2. 剔除空行、纯括号行、import/require 行
3. 统计注释行数（// 和 /* */）
4. 计算比例，判断是否 ≥ 30%（10 行中至少 3 行注释）

**第二遍 — 检查注释准确性：**
1. 阅读注释 → 理解注释描述的意图
2. 阅读对应的代码 → 验证代码实际做的事情
3. 标记不匹配的注释

**第三遍 — 小白视角评估：**
1. 是否使用了小白可能不懂的专业术语？是否有解释？
2. 函数级别的注释是否说明了"为什么"而不是"是什么"？
3. 复杂逻辑是否有分段说明？
4. 边界条件和特殊值是否有说明？

### 步骤 3：输出检查报告

使用以下格式输出报告：

```markdown
## 📋 注释检查报告

### 📁 文件：src/xxx.ts

| 检查项 | 状态 | 说明 |
|--------|:----:|------|
| ① 注释密度 | ✅ / ⚠️ / ❌ | 共 X 行，注释 Y 行，占比 Z% |
| ② 注释准确性 | ✅ / ⚠️ / ❌ | 发现 X 处不匹配 |
| ③ 小白可读性 | ✅ / ⚠️ / ❌ | 发现 X 处难以理解 |

#### 具体问题列表

1. **第 42 行** — ❌ 注释过时
   - 注释说："根据用户等级计算折扣"
   - 实际代码：只判断了是否为VIP，没有等级计算

2. **第 78 行** — ⚠️ 注释不足
   - 一段 15 行的复杂计算没有一个注释
   - 建议：分段说明每一步的计算意图

#### 改进建议

- 为 `calcTotal` 函数添加 JSDoc 注释，说明参数和返回值
- 第 88-95 行的循环建议添加注释说明迭代逻辑
```

### 严重程度判定

| 等级 | 含义 | 图标 |
|:----:|------|:----:|
| ❌ 违规 | 需要修复 | 注释缺失、注释错误、注释误导 |
| ⚠️ 警告 | 建议优化 | 注释不够充分、可读性一般 |
| ✅ 合格 | 符合标准 | 注释充足准确且易懂 |
